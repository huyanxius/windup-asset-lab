# Windup 演示模式工程手册

## 目标

当前产品只保证稳定、零外部依赖的演示闭环：

```text
选择来源 → 定义角色或动作 → 创建演示任务 → 轮询候选
→ 显式采用 → 本地演示资产库 → 逐帧审核 → 导出/Cocos 预览
```

不提供 API Key、Provider 会话、付费模型或远程生成地址。

## 状态与保底

| 数据 | 所有者 | 首选存储 | 保底 |
|---|---|---|---|
| 演示角色、任务、采用结果、审核版本 | `demo-api-client.js` | `localStorage` | 当前页面内存 |
| 内建角色和动作帧 | `character-catalog.js` | 打包静态资源 | 始终可读 |
| 编辑器选择与逐帧状态 | `EditorSession` / `ReviewStore` | 页面内存与本地存储 | 默认少年 |
| 播放和移动 | motion reducer / `PlaybackClock` | 页面内存 | 重置到待机 |

保底顺序固定为：

1. 读取浏览器本地演示状态。
2. 存储被禁用或写入失败时，保留当前页面内存状态。
3. 状态缺失、损坏或指定角色不存在时，回退到打包内置角色并显示提示。

## 故障处理

1. 页面不得等待外部网络或端口。
2. 浏览器不得调用 `fetch`；架构检查会直接失败。
3. 演示任务保持 `queued → generating → processing → awaiting_review → approved`，便于验证真实交互节奏。
4. 本地状态损坏时不显示永久 loading，直接启用内置资产。
5. 候选只有在用户点击采用后才进入演示资产库。
6. 任何回退都应有可见状态文字，不能静默卡死。

## 每次迭代

```text
git status → 小范围实现 → Demo API 单测 → 全量前后端测试
→ 架构边界 → JavaScript/Python 语法 → git diff --check
```

视觉验收重点：项目资产、创建角色、动作生成、审核台均可在没有后端 API 的情况下打开；主按钮能推进状态；刷新后本地演示角色可恢复。
