# Architecture Decisions

这些决策是后续改动的默认前提。若要反转，必须在 PR 中写清原因、迁移方案和兼容影响。

## ADR-001 · 产品契约单一来源

- **决定：** `contracts/windup.v1.json` 是动作、视角、FPS、循环、相位和模型的唯一来源。
- **原因：** 防止前后端分别扩展导致生成、审核和播放语义漂移。
- **结果：** JS 常量/类型与 Python 常量自动生成；CI 检查漂移。

## ADR-002 · 候选资产先审核再采用

- **决定：** 生成结果只进入 job 目录；明确 promote 后才备份并写正式目录。
- **原因：** 模型输出不稳定，自动质检也无法判断完整语义。
- **结果：** 生成、审核、正式交付可以失败和重试，而不会破坏现有资产。

## ADR-003 · API Key 会话隔离且不持久化

- **决定：** 页面只提交一次 Key；后端按 HttpOnly 会话标识保存在进程内存，job 仅接收凭据快照。
- **原因：** 避免浏览器存储、任务文件和多用户全局覆盖泄露密钥。
- **结果：** 服务重启后需要重新连接；生产账户系统可替换 `ProviderSessionStore`。

## ADR-004 · MVP 使用文件 Store，但上层只依赖接口

- **决定：** job 与 review 当前使用原子 JSON 文件和线程锁。
- **原因：** 本地比赛原型无需数据库运维，同时需要恢复与并发保护。
- **结果：** 达到规模阈值后替换 Store，不改变路由和页面用例。

## ADR-005 · 状态所有权优先于框架

- **决定：** 继续使用小型 ES Modules；用 `EditorSession`、纯 reducer、controller 和生成类型明确边界。
- **原因：** 当前复杂度不需要引入大型前端框架，问题本质是状态竞争而不是渲染库。
- **结果：** 当跨页面领域模型显著增长时可迁移 TypeScript，但迁移不改变产品契约和用例边界。

## ADR-006 · 完整动作优先生成动作条，单帧独立修复

- **决定：** 完整 8 帧动作默认一次生成横向动作条并确定性切分；被退回的帧继续使用独立生成。
- **原因：** 初次逐帧生成需要 8 次模型调用，耗时更长且角色风格易漂移；动作条共享一次构图上下文。
- **结果：** 完整动作通常由 8 次调用降为 1 次，同时保留局部修复能力；动作条格式异常可回退逐帧，但供应商错误直接失败。

## ADR-007 · 目录、执行与正式发布是独立边界

- **决定：** `AssetCatalog` 只读取正式资产，`GenerationExecutor` 只执行候选任务，`AssetPublisher` 只负责带备份的正式采用。
- **原因：** 把目录扫描、模型调用、任务状态和文件覆盖堆在应用服务中，会让任何新流程同时改动多个风险域。
- **结果：** `GenerationApplication` 只做校验与编排；新角色在临时目录完整组装后原子入库，动作采用失败时恢复备份。

## ADR-008 · 工作流模板只复用配置，不复用任务或候选资产

- **决定：** 已验证画布可保存项目约束、节点顺序、动作描述与 FPS；每次复用必须创建新 character job。
- **原因：** 重用旧 job 或候选图片会破坏溯源、质检和正式资产边界，也可能误带用户输入或凭据。
- **结果：** `WorkflowTemplateStore` 与 `JobStore` 分离；模板运行记录 ID/版本来源，生成结果仍在 `awaiting_review` 等待明确 promote。
