# Windup 生产链路审计与改造报告

## 结论

本次已打通“创建角色 → 生成基础动作 → 自动切帧 → 候选审核 → 原子入库 → 角色管理 → 审核台读取”的完整链路。新角色不再只保存一张母版，完整动作也不再默认逐帧调用模型。

## 审计发现

1. 原创建任务只生成并复制 `base.png`，动作生成属于另一个任务，因此新角色入库后必然出现“尚无动作”。
2. `full` 模式实际循环调用模型 8 次，一致性和耗时都不可控。
3. 仓库早期已有横向动作条、抠图、八等分和归一化工具，但没有接入在线生成服务。
4. 队友仓库当前同样以逐帧生成作为主路径，并未实现在线动作条生成；本次复用的是本仓库原型中已验证的切帧思想。
5. 应用服务同时承担目录、生成、进度、溯源和覆盖正式文件，职责过多。

## 新流程

```text
角色定义
  → 1 次角色母版生成
  → 1 次待机动作条生成 → 抠图 → 8 等分 → 归一化
  → 1 次行走动作条生成 → 抠图 → 8 等分 → 归一化
  → 几何连续性质检
  → 整体人工确认
  → 临时目录完整组装
  → 原子写入正式角色库
  → 角色管理与审核台直接读取
```

默认角色包由 17 次模型调用降为 3 次；单个完整动作由 8 次降为 1 次。模型若返回不可切分的动作条，会回退到逐帧生成；鉴权、配额和网络错误不会被错误地吞掉。坏帧仍可用单帧模式独立修复。

## 架构结果

- `GenerationApplication`：输入校验与用例编排。
- `GenerationExecutor`：后台执行、进度与溯源。
- `ActionPipeline`：动作条与单帧双路由。
- `AssetCatalog`：正式角色与动作清单。
- `AssetPublisher`：候选校验、原子角色包入库、动作备份与失败回滚。
- `processing.py`：确定性抠图、切帧、归一化与几何连续性。
- `contracts/windup.v1.json`：生成路由与基础动作包的唯一版本化定义，版本 `1.1.0`。

## 已验证

- Demo HTTP 完整角色包：`queued → generating → awaiting_review → approved`。
- 入库结果包含 `side/idle` 8 帧和 `side/walk` 8 帧，角色 API 可立即读取。
- 完整动作任务返回 8 帧、`generationRoute=sheet` 与调用次数记录。
- 前端 22 项、后端 8 项与架构边界检查通过。
- 未使用浏览器截图自动化；未提交 API Key、任务数据或用户持有的 Skeleton 帧。

## 仍需人工判断

几何质检能够发现空帧、主体高度、水平中心和脚底线问题，但无法可靠判断腿是否真正交叉、解剖是否正确、角色是否变脸。动作条结果仍需逐帧人工审核；生产环境还需把本地后台线程升级为队列和对象存储。
