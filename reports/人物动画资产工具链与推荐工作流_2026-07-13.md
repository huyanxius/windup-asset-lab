# 人物动画资产工具链与推荐工作流

## 1. 最值得保留的工具

### A 级：应当产品化

#### 1. 角色母版与生成批次

**作用：** 在多动作、多视角生成时保持角色身份。

母版至少应固定：

- 正面、侧面、背面、俯视和 2.5D 参考。
- 头身比、发型轮廓、脸型和年龄感。
- 服装分区、主色、配饰位置和材质。
- 母版版本、参考图版本和生成批次。

**价值：** 这不是一张“好看的设定图”，而是后续生成、审核和追溯的身份基准。

#### 2. 色键抠图器

**当前工具：** `remove_chroma_key.py`

**推荐能力：**

- 自动从图片边缘推断背景色。
- 软遮罩，保留头发和围巾细节。
- 边缘收缩，减少色边。
- Despill，去除绿色反射污染。
- 可配置透明和不透明阈值。

**价值：** 色边、半透明残留会直接进入引擎，是非常常见的交付故障。

#### 3. 比例切帧与归一化器

**当前工具：** `tools/process-action-sheet.js`

**已验证的正确做法：**

- 按大图宽度的比例计算每格边界，不假设宽度能被帧数整除。
- 基于 Alpha 轮廓裁掉空白。
- 像素风使用最近邻缩放。
- 统一输出 256×256。
- 对 locomotion 类动作统一人物大小、中心和脚底锚点。

**必须改进：** 不能用同一套“固定脚底线”处理所有动作。`jump`、击飞和倒地需要保留纵向位移，应使用独立 profile。

#### 4. 命令行动画质检器

**新增工具：** `tools/audit-animation.js`

**检查内容：**

- 帧数、画布尺寸、Alpha 透明背景。
- 脚底线漂移、主体高度漂移。
- 相邻帧质心位移尖峰。
- 相邻帧轮廓面积波动。
- 循环动作的首尾接缝。
- 根据 `locomotion`、`jump`、`oneshot` 使用不同规则。

**输出：**

- Markdown：给人阅读和审核。
- JSON：给平台、流水线或 CI 读取。
- 超出阈值时返回非零状态，可作为自动门禁。

**使用示例：**

```bash
NODE_PATH=/Users/huyan/.cache/codex-runtimes/codex-primary-runtime/dependencies/node/node_modules \
node tools/audit-animation.js \
  assets/resources/character/frames walk \
  --loop --profile locomotion --fps 8 \
  --out reports/qc-walk-side
```

**已验证结果：**

- `side / walk / 8 帧`：100/100，通过。
- `side / jump / 8 帧`：67/100，需处理。检出相邻帧位移最大 23.8px、轮廓面积最大波动 42.8%。这说明跳跃帧仍存在跳变和轮廓不稳定，不应直接进入游戏交付。

#### 5. 逐帧审核台与导出门禁

**当前工具：** `asset-lab/`

**关键点：**

- 自动质检只是第一道门。
- 每帧必须能够单独通过、退回和写备注。
- 审核状态与批次绑定。
- 所有帧通过后才能导出。

**价值：** 自动指标不能判断脚步是否反了、角色是否变脸、动作是否真正可读。

#### 6. 真实引擎联调桥

**当前方式：** Asset Lab 通过 `postMessage` 向 Cocos 发送视角、动作、循环状态和 FPS；Cocos 动态加载真实帧资产并返回结果。

**价值：** 资产管理界面不自己模拟一个假游戏，最终判断以真实引擎为准。

### B 级：非常实用的辅助工具

- **Sharp**：裁切、Alpha 处理、最近邻缩放、合成和 PNG 导出。
- **Canvas 像素扫描**：在浏览器内实时计算包围盒、质心和面积。
- **Cocos `resources.loadDir`**：按视角和动作动态加载资源。
- **Cocos CLI Build**：每次修改后构建 Web 版，防止“代码看起来对，引擎跑不起来”。
- **Python HTTP Server**：快速运行平台和已构建游戏。
- **浏览器自动化**：验证播放、切换、人物移动、Cocos 同步和 JavaScript 错误。
- **`node --check`**：低成本脚本语法检查。
- **`unzip -t`**：交付包完整性验证。

## 2. 推荐的标准工作流

### 阶段 0：定义交付契约

在生成之前明确：

- 角色 ID 与母版版本。
- 视角、方向、动作名、帧数、FPS 和是否循环。
- 画布、锚点、输出命名和引擎目录。
- 动作 profile：`locomotion`、`jump` 或 `oneshot`。

**门禁：** 交付契约不完整，不开始生成。

### 阶段 1：生成原始动作图

1. 引用锁定的角色母版。
2. 在提示中明确列出动作相位，不只写“行走 8 帧”。
3. 固定镜头、人物大小、身份特征和背景色。
4. 保留原始输出，使用新批次号，不覆盖历史。

**门禁：** 原始图中若已经出现角色变脸、帧缺失或明显错误，直接重生成，不进入后处理。

### 阶段 2：确定性后处理

1. 色键抠图。
2. 按比例切帧。
3. 主体轮廓检测。
4. 根据 action profile 做尺寸和锚点归一化。
5. 输出标准透明 PNG。

**门禁：** 必须保留原始大图、透明大图和最终帧，不能只留最后结果。

### 阶段 3：自动工程质检

运行 `audit-animation.js`，生成 Markdown 和 JSON。

**门禁：** 任一必检项失败，不进入人工审核和引擎。

### 阶段 4：人工视觉审核

人工必须检查：

- 脸、发型、服装、配色和体型是否稳定。
- 左右脚是否真正交替。
- 解剖是否正确，手脚有无粘连或反向。
- 动作相位是否完整。
- 循环播放是否存在闪跳。
- 风格、情绪和动作可读性是否符合项目。

**门禁：** 逐帧通过后才生成导出包。

### 阶段 5：真实引擎验证

1. 将当前动作推送到 Cocos。
2. 在真实场景中观察大小、锚点、速度、方向和循环。
3. 用 A / D 或自动巡走进行连续运动测试。
4. 切换横屏、俯视和 2.5D，检查资产是否真正替换。

**门禁：** 以引擎中的可见效果为准，不以平台状态文字作为成功证据。

### 阶段 6：交付与追溯

交付包应包含：

- 通过审核的透明 PNG 帧。
- sprite sheet / atlas。
- metadata：角色、视角、动作、FPS、循环、锚点和帧序。
- 自动质检报告。
- 人工审核结果和退回记录。
- 生成批次、母版版本和参考图版本。
- 引擎验证结果。

## 3. 不应自动化或不能信任的部分

- 不能仅凭质心和轮廓判定“走路正确”。
- 不能将工程质检 100 分解读为美术质量 100 分。
- 不能用高 FPS 掩盖中间相位缺失。
- 不能用同一套归一化规则处理行走、跳跃、倒地和击飞。
- 不能用 CSS 旋转或压缩代替真实多视角资产。
- 不能覆盖原始生成图，否则无法回溯和改进。

## 4. 近期应补的工具

1. **Action Profile 处理器**：为 locomotion、jump、airborne、knockdown 使用不同锚点和位移规则。
2. **跨视角一致性检测器**：对头高、发型轮廓、服装区域、主色和配饰锚点打分。
3. **脚步语义检测器**：识别左右脚并验证交替、支撑相和腾空相。
4. **单帧定向重生成器**：退回单帧后自动携带前后帧与母版重生成。
5. **标准导出器**：一次生成 PNG、atlas、metadata、预览 GIF 和 Cocos 导入配置。
6. **批次可观测仪表盘**：自动记录提示词、参考图、模型、耗时、成本、失败和重试。

## 5. 结论

最好的单个工具不是“图片生成器”，而是能将原始生成结果稳定地变成引擎资产的工具链。

推荐的产品闭环是：

> 角色母版 → 动作规格 → 生成批次 → 确定性后处理 → 自动工程质检 → 人工逐帧审核 → 真实引擎验证 → 导出与追溯。

其中任何一环被省略，都可能把不稳定的 AI 输出带入游戏项目。
